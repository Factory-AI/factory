カスタムドロイドは、Markdownで定義された再利用可能なサブエージェントです。各ドロイドは独自のシステムプロンプト、モデル設定、ツールポリシーを持っているため、コードレビュー、セキュリティチェック、リサーチなどの集中したタスクを、指示を再入力することなく委任できます。

---

## 1 · カスタムドロイドとは？

カスタムドロイドは、プロジェクトの `.factory/droids/` ディレクトリまたは個人の `~/.factory/droids/` ディレクトリ内の `.md` ファイルとして存在します。CLIはこれらのフォルダ（最上位レベルのファイルのみ）をスキャンし、各定義を検証し、**Task**ツールの `subagent_type` ターゲットとして公開します。これにより、プライマリアシスタントはセッション中に特定目的のヘルパーを起動できます。

<Note>
  カスタムドロイドはデフォルトで有効になっています。必要に応じて、設定（`/settings`）の実験的機能セクションで無効にできます。
</Note>

- **プロジェクトドロイド**は `<repo>/.factory/droids/` に配置され、チームメイトと共有されます。
- **個人ドロイド**は `~/.factory/droids/` に配置され、ワークスペース間で引き継がれます。
- 名前が一致する場合、プロジェクト定義が個人定義を上書きします。

---

## 2 · なぜ使うのか？

- **高速な委任** – 複雑なチェックリストを一度エンコードし、単一のツール呼び出しで再利用できます。
- **厳密な安全性** – エージェントを読み取り専用、編集専用、または厳選されたツールセットに制限できます。
- **コンテキストの分離** – 各サブエージェントは新しいコンテキストウィンドウで実行され、プロンプトの肥大化を回避します。
- **反復可能なレビュー** – チーム固有のレビュー、テスト、リリースゲートをバージョン管理可能なコードとして記録できます。

---

## 3 · クイックスタート

1. `/droids` を実行してドロイドメニューを起動します。
2. **Create a new Droid**を選択し、保存場所（プロジェクトまたは個人）を選択し、ウィザードに従って以下を設定します：
   - ドロイドが行うべきことの説明
   - システムプロンプト（自動生成または手動編集）
   - 識別子（ドロイドの名前）
   - モデル（または親セッションから継承）
   - ツール（ツールIDの明示的リストまたはカテゴリ）
3. 保存します。CLIは選択した `droids/` ディレクトリに `<name>.md` を書き込み、ファイル名を正規化します（小文字、ハイフン区切り）。
4. ドロイドに使用を指示します。例：「`code-reviewer` サブエージェントでTaskツールを実行してこのdiffをレビューして」や、自動化からのトリガー。

ドロイドファイルの変更は、次回のメニューオープンまたはTaskツール呼び出し時に反映されます。

---

## 4 · 設定

各ドロイドファイルは、YAMLフロントマターを含むMarkdownです。```md
---
name: code-reviewer
description: Focused reviewer that checks diffs for correctness risks
model: inherit
tools: read-only
---

You are the team's senior reviewer. Examine the diff the parent agent shares and:

- flag correctness, security, and migration risks
- list targeted follow-up tasks if changes are required
- confirm tests or manual validation needed before merge

Respond with:
Summary: <one-line finding>
Findings:

- <bullet>
- <bullet>
```ツールをより細かく制御するために、配列として指定することもできます：```md
---
name: deep-analyzer
description: Thorough analysis with extended thinking
model: claude-sonnet-4-5-20250929
reasoningEffort: high
tools: ["Read", "Grep", "Glob", "WebSearch"]
---

Perform deep analysis of the code or problem presented...
```主要なメタデータフィールド：

| フィールド         | 備考                                                                                                                          |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| `name`            | 必須。小文字、数字、`-`、`_`。`subagent_type`の値とファイル名を決定します。                                  |
| `description`     | オプション。UIリストに表示されます。500文字以内に収めてください。                                                                               |
| `model`           | `inherit`（親セッションのモデルを使用）、またはモデル識別子を指定。ビルトインモデルの場合、`claude-sonnet-4-5-20250929`などの値を使用。カスタムモデル（BYOK）の場合、設定の`model`フィールドに`custom:`を付けたもの（例：`custom:gpt-4o-mini`）を使用し、`model_display_name`では**ありません**。モデルIDについては[料金プランページ](/jp/pricing#available-models)を参照してください。 |
| `reasoningEffort` | オプション。対応するモデルの推論努力レベルを設定（例：`low`、`medium`、`high`）。`model`が`inherit`の場合は無視されます。選択したモデルと互換性がある必要があります。 |
| `tools`           | ツールの選択：すべてのツールの場合は省略、カテゴリ文字列（例：`read-only`）を使用、または`["Read", "Edit", "Execute"]`のようなツールIDの配列を指定。大文字小文字を区別します。 |

プロンプトは最低でも`name`を含むYAMLフロントマターで始まり、空でない本文を含む必要があります。`DroidValidator`はエラー（無効な名前、未知のモデル、未知のツール）と警告（説明の欠如、重複するツール）を表面化します。検証問題は、ファイルの読み込みに失敗した際にCLIログに表示されます。

### ツールカテゴリ → 具体的なツール

`tools`値として直接カテゴリ名（例：`tools: read-only`）を使用するか、配列で個別のツールIDを指定できます。

| カテゴリ    | ツールID                         | 目的                            |
| ----------- | -------------------------------- | ---------------------------------- |
| `read-only` | `Read`、`LS`、`Grep`、`Glob`     | 安全な分析とファイル探索 |
| `edit`      | `Create`、`Edit`、`ApplyPatch`   | コード生成と修正  |
| `execute`   | `Execute`                        | シェルコマンド実行            |
| `web`       | `WebSearch`、`FetchUrl`          | インターネット調査とコンテンツ      |
| `mcp`       | 動的に追加（存在する場合）   | Model Context Protocolツール       |

明示的な配列では、上記のツール名を使用する必要があります（大文字小文字を区別）。未知の名前は検証エラーを引き起こします。

<Note>
  `TodoWrite`はタスク追跡を有効にするため、すべてのドロイドに自動的に含まれます。ツールリストに追加する必要はありません。
</Note>

<Note>
  OpenAIモデルで`Edit`を使用する場合、互換性のため`ApplyPatch`が自動的に含まれます。`model`が`inherit`の場合、モデルプロバイダー間でのカバレッジを確保するため、両方のツールが有効になります。
</Note>

## 5 · UIでのドロイドの管理

`/droids`でモーダルが開き、以下が表示されます：

- **ドロイドのリスト** – 各ドロイドを以下の情報と共に表示：
  - 名前と括弧内のモデル
  - 説明のプレビュー
  - 場所のバッジ（Project / Personal）
  - ツールの概要（例：「すべてのツール」または選択されたツールの数）
- **新しいドロイドを作成** – ガイド付きウィザードを起動：
  1. 場所を選択（ProjectまたはPersonal）
  2. ドロイドに何をさせるかを説明
  3. システムプロンプトを生成または手動編集
  4. 識別子、モデル、ツールを確認
- **Claude Codeからインポート** – `~/.claude/agents/`から既存のエージェントをカスタムドロイドとしてインポート
- **アクション** – ドロイドの表示、編集、削除、またはリロードしてリストを更新

## 5.5 · Claude Codeサブエージェントのインポート

Claude Codeで作成したエージェントをカスタムドロイドとしてインポートできます。これにより、既存のClaude Codeエージェントをドロイドシステムで再利用できます。

### インポートの方法

1. `/droids`を実行してドロイドメニューを開く
2. **I**を押してインポートフローを開始
3. CLIがClaude Codeエージェントディレクトリをスキャン：
   - **プロジェクトスコープ**：`<repo>/.claude/agents/`（ワークスペース固有のエージェント）
   - **パーソナルスコープ**：`~/.claude/agents/`（個人のエージェント）
4. 利用可能なエージェントのリストを確認：
   * `(already exists)`マークのエージェントはデフォルトで非選択状態
   * まだドロイドにインポートされていないエージェントは事前選択済み
5. **Space**で個別選択の切り替え、**A**ですべての切り替え
6. **Enter**で選択されたエージェントをインポート

### インポート中に起こること

インポート処理はClaude Codeエージェントをドロイドに変換します：

1. **メタデータ抽出**：
   - エージェント名 → ドロイドの`name`
   - エージェントの説明（例や使用方法を含む） → ドロイドの`description`
   - エージェントの指示 → ドロイドシステムプロンプト（本文）

2. **設定のマッピング**：
   - モデル：Claude CodeモデルファミリーをFactoryモデルにマッピング：
     - `inherit` → `inherit`
     - `sonnet` → 利用可能な最初のSonnetモデル
     - `haiku` → 利用可能な最初のHaikuモデル
     - `opus` → 利用可能な最初のOpusモデル
   - ツール：Claude CodeツールをFactoryツールにマッピング（ツールがマッピングされない場合は検証警告が表示される可能性）
   - 場所：デフォルトでPersonal `~/.factory/droids/`にインポート

3. **ツール検証**：
   - 一部のClaude CodeツールはFactoryに相当するものがない場合があります
   - 無効なツールは警告とともに一覧表示されます：「無効なツール：[一覧]」
   - ドロイドを編集してツールマッピングを修正したり、ツールアクセスを調整できます

4. **ファイル作成**：
   - `~/.factory/droids/`（個人の場所）に`.md`ファイルを作成
   - ファイル名は正規化されます（小文字、ハイフン区切り）
   - ファイル形式：YAMLフロントマター + システムプロンプト本文

5. **インポートレポート**：
   - 各エージェントの成功/失敗を表示
   - インポートされたエージェントはドロイドリストですぐに利用可能
   - インポートされたドロイドを編集してモデル、ツール、またはプロンプトを調整できます

### インポートフローの例

**選択画面**（インポート前）：```
Import Droids (.claude/agents)

Project (<repo>/.claude/agents/):
> [x] polite-greeter
  Use this agent when the user requests a greeting, says hello, or asks for a polite
  welcome message. Examples: <example>...

  [ ] code-summarizer
  Use this agent when you need to understand what a code file does without diving into
  implementation details. Examples: <example>...

Personal (~/.claude/agents/):
  [x] security-checker
  Security analysis agent...

↑/↓ navigate • Space select • A toggle all • Enter import • B back
```**インポート後**（Droidリストに戻る）：```

> code-reviewer (gpt-5-codex)
  This droid verifies the correct base branch and committed...
  Location: Project  •  Tools: All tools

  polite-greeter (claude-opus-4-5-20251101)
  Use this agent when the user requests a greeting, says he...
  Location: Personal  •  Tools: 1 selected

  code-summarizer (claude-sonnet-4-5-20250929)
  Use this agent when you need to understand what a code fi...
  Location: Personal  •  Tools: All tools
```### ツール検証エラーの処理

インポート後に **"Invalid tools: [リスト]"** が表示された場合、一部のClaude Codeツールに対応するFactoryツールが存在しないことを意味します:

1. **droidを表示** (Enterキーを押す) してツールの全リストを確認
2. **droidを編集** (Eキーを押す) して調整:
   - リストから無効なツールを削除
   - 有効なFactoryツールのみを保持
3. **利用可能なツールを確認** - リストに利用可能なFactoryツールが表示されます

一般的にマッピングされないツール:

* `Write`、`NotebookEdit`、`BrowseURL`などのClaude CodeツールはFactoryには存在しません
* 同等のFactoryツールに置き換えてください:
  * `Write` → `Edit`、`Create`
  * `BrowseURL` → `WebSearch`、`FetchUrl`
* または`tools`セクションを完全に削除してすべてのFactoryツールを有効にしてください

## 6 · カスタムdroidを効果的に使用する

- **Taskツール経由で呼び出し** – droidはカスタムdroidを自動的に呼び出すか、直接リクエストできます（「この変更にサブエージェント`security-auditor`を使用してください」）。
- **モデルを戦略的に選択** – 親セッションと一致させるために`inherit`を使用するか、専門的なタスクに異なるモデルを指定:
  - 簡単な分析とサマリータスクには小さい/高速なモデル（低コスト）。
  - 複雑な推論、コードレビュー、マルチステップ分析には大きい/高機能なモデル。
  - 利用可能なモデルIDについては[プランページ](/jp/pricing#available-models)を参照してください。
- **ツールアクセスを制限** – 明示的なツールリストを使用してサブエージェントができることを制限し、予期しないシェルコマンドやその他の危険な操作を防ぎます。
- **ライブアップデートを活用** – Taskツールはライブプログレスをストリーミングし、サブエージェントの実行中にツール呼び出し、結果、TodoWriteアップデートをリアルタイムで表示します。
- **出力を構造化** – `Summary:`や`Findings:`などのセクションを出力するようにプロンプトを整理し、TaskツールUIが結果を明確にサマリーできるようにします。
- **共有と協力** – チームメイトが共有droidを使用できるように`.factory/droids/*.md`をリポジトリにチェックインし、コードと同じようにプロンプトアップデートをバージョン管理します。
- **Claude Codeエージェントを活用** – 既存のClaude Codeエージェントをインポート（セクション5.5を参照）してFactoryでカスタムdroidとして再利用します。

## 7 · 例

### コードレビューア（プロジェクトスコープ）```md
---
name: code-reviewer
description: Reviews diffs for correctness, tests, and migration fallout
model: inherit
tools: ["Read", "LS", "Grep", "Glob"]
---

You are the team's principal reviewer. Given the diff and context:

- Summarize the intent of the change.
- Flag correctness risks, missing tests, or rollback hazards.
- Call out any migrations or data changes that need coordination.

Reply with:
Summary: <one-line>
Findings:

- <issue or ✅ No blockers>
  Follow-up:
- <action or leave blank>
```Use: "ステージングされた差分に対してサブエージェント `code-reviewer` を実行する。"

### セキュリティスイーパー（個人スコープ）```md
---
name: security-sweeper
description: Looks for insecure patterns in recently edited files
model: inherit
tools: ["Read", "Grep", "WebSearch"]
---

Investigate the files referenced in the prompt for security issues:

- Identify injection, insecure transport, privilege escalation, or secrets exposure.
- Suggest concrete mitigations.
- Link to relevant CWE or internal standards when helpful.

Respond with:
Summary: <headline>
Findings:

- <file>: <issue>
  Mitigations:
- <recommendation>
```### タスクコーディネーター（ライブ進捗表示付き）```md
---
name: task-coordinator
description: Coordinates multi-step tasks with live progress updates
model: inherit
tools: ["Read", "Edit", "Execute"]
---

You are a task coordinator. Break down the goal into actionable steps:

1. Use TodoWrite to create and update a task list
2. For each task, read relevant files and execute commands as needed
3. Report progress in real-time using TodoWrite updates

Keep the task list updated with completion status (pending, in_progress, completed).
```---

カスタムDroidを使用すると、部族知識をコードとして捉えることができます。特化したプロンプトを一度作成し、適切なツールとモデルを割り当て、主要なアシスタントが重い作業をあなたが設計したサブエージェントに委任できるようにします。
