## 概要

設定済みのSlackチャンネルにアラートが届くと、Incident Response 自動化が関連付けられたDroidセッションを自動で開始し、適切なコンテキスト、コンピューター、指示を使ってDroidがインシデントを調査できるようにします。

DroidはRCAの実施、必要に応じた修正PRの作成、将来同じ種類のアラートをより速く調査できるようにするインシデントrunbookの更新まで進められます。

## クイックスタート: Factory App からセットアップする

Incident Response は Factory App の [Automations](/jp/web/automations) で設定します。

<Steps>
  <Step title="Factoryをインシデントチャンネルに招待する">
    Slackで、アラートbotがインシデントを投稿するチャンネルを開き、次を実行します。

    ```text
    /invite @Factory
    ```

    まだSlackを接続していない場合は、先に[Slack連携](/jp/integrations/slack)を設定してください。チャンネルピッカーには、Factoryが招待されているチャンネルだけが表示されます。
  </Step>

  <Step title="新しい自動化を作成する">
    Factory App のサイドバーで **Software Factory** 配下の [**All Automations**](https://app.factory.ai/automations) を開き、**New automation** をクリックします。
  </Step>

  <Step title="Incident Response テンプレートを選ぶ">
    テンプレートギャラリーで、Software Factory templates 配下の **Incident Response** テンプレートを選択します。

    <Frame>
      <img src="/images/incident-response-templates.png" alt="Incident Response テンプレートを含む自動化テンプレートギャラリー" />
    </Frame>
  </Step>

  <Step title="自動化に名前を付け、トリガーチャンネルを選ぶ">
    自動化に名前を付け、**Triggers** でインシデントチャンネルを検索して選択します。**Trigger on messages from** の既定値は **Bots only** で、インシデントを投稿するアラートbotに適しています。人間の投稿でも調査を開始したい場合にのみ変更してください。

    <Frame>
      <img src="/images/incident-response-create.png" alt="新しい Incident Response 自動化のフォーム" />
    </Frame>
  </Step>

  <Step title="アカウントとコンピューターを選ぶ">
    - **Run as** は、自動化をどのIDと課金先で実行するかを設定します。インシデント対応では、コンピューターとセッションの操作性をチームで共有できるよう、[サービスアカウント](/jp/enterprise/service-accounts) の使用を推奨します。
    - **Run on** は、Droidが調査を実行するコンピューターを設定します。適切なリポジトリ、観測ツール、その他エージェントの助けになるものが揃ったコンピューターを選んでください。Droid Computersについて詳しくは[こちら](/jp/cli/features/droid-computers)を参照してください。
  </Step>

  <Step title="MCPサーバーを設定する">
    **Configure MCP Servers** を使って、Droidがインシデントに対応するために必要な観測ツールやコンテキストソースを追加し、認証します。MCPサーバーはコンピューターごとに設定され、**設定にはそのコンピューターへの接続が必要**です。問題が起きる場合は、数回再試行するかコンピューターを変更してください。
  </Step>

  <Step title="プロンプトとモデルを確認する">
    テンプレートには、チャンネルからトリガーされたすべてのセッションに注入される次のプロンプトが事前入力されています。

    ```text
    Use the `/incident` skill for Slack incidents and run an RCA if applicable.
    If the thread message is an incident or asking for RCA/root-cause analysis, invoke the `incident` skill before investigating.
    ```

    デフォルトのまま使うか、チャンネル固有の指示を追加し、必要に応じてモデルを選択してください。
  </Step>

  <Step title="セッションの公開範囲を設定して作成する">
    トリガーされたセッションを **Team** と **Private** のどちらで表示するかを選び、必要に応じて **Additional Settings** を確認してから、**Create** をクリックします。
  </Step>

  <Step title="テストアラートを送信する">
    実際のインシデント送信に使うのと同じ種類のbotから、トップレベルのテストアラートを投稿します。自動化がDroidセッションを開始し、Slackスレッドからそのリンクを確認できることを確かめてください。実行結果は自動化の詳細ビューから確認できます。
  </Step>
</Steps>

## 設定の詳細

### プロンプトをカスタマイズする

自動化プロンプトは、チャンネルからトリガーされるすべてのセッションに注入されます。内容はインシデント対応に特化させてください。

- 適切な場合はRCAを実施し、`/incident`ワークフローを使うようにDroidに伝える
- そのチャンネルに対応する主要なサービスやリポジトリを示す
- runbook、ダッシュボード、一般的なアラートソースを案内する
- 不確実性を要約して共有するタイミングなど、エスカレーションの期待値を明示する

自動化を汎用的なSlackの入口にしてしまうようなプロンプトは避けてください。インシデント対応は、プロンプトが狭く運用に特化しているほど効果を発揮します。より広いSlackワークフローが必要な場合は、代わりに別の [Slack automation](/jp/web/automations) を作成してください。

### チャンネルの選択とトリガー送信元

インシデント対応は、インシデントアラートがトップレベルのメッセージとして届くチャンネル向けに設計されています。スレッド返信は新しいセッションの開始には使われないため、通常の会話でDroidが繰り返し起動するのを防げます。

最良の結果を得るには、`#incidents`、`#alerts-production`、またはサービス固有のインシデントチャンネルのような専用チャンネルを使用してください。

チャンネルピッカーには、Factoryが招待されているチャンネルだけが表示されます。**Trigger on messages from** の既定値は **Bots only** なので、通常のチャンネルメッセージで調査が開始されることはありません。人間の投稿でもインシデントセッションを開始できるようにしたい場合にのみ範囲を広げてください。

### セッションの公開範囲とモデル

**Session privacy** は、トリガーされたセッションを誰が閲覧できるかを制御します。**Private** セッションは実行に使うIDにのみ表示されます。**Team** セッションは他のメンバーも確認できるため、インシデントの引き継ぎやポストモーテムレビューに役立ちます。

チームでインシデント分析用に明確な好みがない限り、モデルはデフォルトのまま使ってください。複雑な本番インシデントでは、利用可能であればより推論能力の高いモデルを選んでください。

### 自動化を管理する

自動化は [**All Automations**](https://app.factory.ai/automations) から開いて、最近の実行とステータスを確認できます。権限に応じて、今すぐ実行、一時停止、再開、共有、フォーク、名前変更、編集、削除ができます。インシデントセッションを一時的に止めたい場合は、削除ではなく自動化を一時停止してください。

## インシデントフローの仕組み

設定後、インシデント対応は次の流れで動作します。

1. アラートbotが、設定済みのSlackチャンネルにトップレベルのメッセージを投稿します。
2. 自動化が、その実行ID、コンピューター、セッションの公開範囲、モデル、プロンプトを使ってDroidセッションを開始します。
3. DroidがSlackアラートのコンテキストを受け取り、一致する `incident-guidelines` runbookガイダンスがあるか確認します。
4. Droidは設定済みツールを使って調査し、RCAが適切な場合は `/incident` ワークフローを呼び出すべきです。
5. FactoryはステータスとセッションリンクをSlackに投稿し、Droidは類似インシデント向けに再利用可能な知見を保存できます。

`incident-guidelines` runbookは、チームで再利用する場合は `.factory/skills/incident-guidelines/SKILL.md`、個人で再利用する場合は `~/.factory/skills/incident-guidelines/SKILL.md` にローカルで保存されます。ここには再利用可能な調査ガイダンスを保存し、シークレットや単発のRCA詳細は保存しないでください。

## トラブルシューティング

<AccordionGroup>
  <Accordion title="Slackチャンネルがチャンネルピッカーに表示されない">
    `/invite @Factory`でFactoryをチャンネルに招待してから、自動化のトリガー設定で再度チャンネルを検索してください。プライベートチャンネルでは、明示的にFactoryを招待する必要があります。
  </Accordion>

  <Accordion title="自動化を作成できない">
    自動化で **Run on** コンピューターが選択されていることを確認してください。**Run as** がサービスアカウントに設定されている場合は、そのサービスアカウントが有効で、少なくとも1台のコンピューターを所有している必要があります。
  </Accordion>

  <Accordion title="テストメッセージ後にセッションが始まらない">
    そのメッセージが設定済みチャンネル内のトップレベルアラートであり、スレッド返信ではないこと、さらに送信者が **Trigger on messages from** の設定と一致していることを確認してください。あわせて、自動化が一時停止されていないことも確認してください。
  </Accordion>

  <Accordion title="Droidは起動するが問題を調査できない">
    選択したコンピューターに、適切なリポジトリ、ツール、認証情報があることを確認してください。Droidが最初に使うべきrunbookやダッシュボードへのリンクを、自動化プロンプトに追加してください。
  </Accordion>

  <Accordion title="RCA が一般的すぎる">
    サービス名、アラートソース、ダッシュボードリンク、ログクエリ例、リポジトリパス、期待する RCA 形式など、チャンネル固有の情報をプロンプトに追加してください。
  </Accordion>
</AccordionGroup>
