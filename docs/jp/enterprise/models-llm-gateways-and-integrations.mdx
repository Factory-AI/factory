Factory は、**既にお使いのAIおよび開発者ツール**に接続するように設計されています。このページでは、Droidが使用できるモデルの制御方法、LLMゲートウェイ経由のトラフィックルーティング、そしてMCPサーバー、ドロイド、コマンド、プラットフォーム統合の管理方法について説明します。

---

## モデル管理と階層

モデルアクセスは、Factory全体で使用されている同じ階層設定システムによって管理されます：

- **組織レベル**
  - **許可されたモデルとカテゴリの正式なリスト**を定義します。
  - モデルを明示的に禁止することができ（例：非エンタープライズAPI）、プロジェクトやユーザーによって再有効化されることを防ぎます。
  - ユーザー提供のBYOKキーがまったく許可されるかどうかを決定します。

- **プロジェクトレベル**
  - 組織承認済みセットの上に**追加のモデルを加える**ことができます（例：プロジェクト固有のファインチューニング）が、禁止されたモデルを再有効化することはできません。
  - プロジェクトのデフォルトを設定できます—例：「テストには小型モデル、リファクタリングには大型モデルを使用」。

- **ユーザーレベル**
  - **許可されたセット内で**個人デフォルトを選択できます。
  - 上位レベルで不許可にされたモデルを表示または選択することはできません。

階層が拡張のみであるため、組織レベルでのアップグレードやポリシー変更は、すべての環境で一貫して適用されます。

## LLMゲートウェイ

多くの企業では、認証、ルーティング、ポリシー実行を処理する**LLMゲートウェイ**の背後でモデルアクセスを集約化しています。

Factoryは2つの方法でゲートウェイと連携します：

1. **直接設定されたゲートウェイエンドポイント**
   - `.factory/settings.json`で特定のプロバイダーのゲートウェイベースURLを指定できます。
   - 環境変数で呼び出しをルーティングできます（例：`ANTHROPIC_BASE_URL`、`OPENAI_BASE_URL`）。
   - [BYOK](/jp/cli/byok/overview)経由で設定されたカスタムモデルは、`base_url`とプロバイダー固有の設定を使用してゲートウェイエンドポイントを指定できるため、モデルが組み込みかカスタムかに関わらず、同じゲートウェイポリシーが適用されます。

2. **組織レベルのゲートウェイポリシー**
   - 組織管理者は、すべてのモデルトラフィックが特定のゲートウェイを経由することを要求できます。
   - BYOKを制限して、中央管理されたキーとIDのみが使用されるようにできます。

ゲートウェイを使用する場合、**データの処理と保持ポリシーはゲートウェイと基盤となるプロバイダーのもの**になります；Droidは単に設定されたエンドポイントと認証情報を使用します。

ゲートウェイ対応モデルを含むカスタムモデル設定の具体例については、[カスタムモデル & BYOK](/jp/cli/byok/overview)を参照してください。これは`~/.factory/settings.json`の`customModels`配列と、それらのモデルが`/model`セレクターにどのように表示されるかを説明しています。

## クラウドプロバイダーとBYOK

Factoryは複数のモデルデプロイメントパターンをサポートします：

- **直接クラウドプロバイダー** – 企業の認証情報を使用してOpenAI、Anthropic、Google、その他を呼び出し。
- **クラウドAIプラットフォーム** – AWS Bedrock、GCP Vertex、Azure OpenAI、IAM対応認証を使用。
- **オンプレミス/セルフホスト型モデル** – 承認されたゲートウェイ経由で公開される、ネットワーク内で動作するモデル。

Bring-your-own-key（BYOK）はポリシーによって制御されます：

- 組織管理者は**ユーザー提供のAPIキー**を許可またはブロックできます。
- BYOKが許可されている場合でも、組織はキーが対象とできるプロバイダーとエンドポイントを制限できます。
- プロジェクト設定では、セキュアストアでチーム全体のモデル用の共有キーや認証情報を定義できます。

高セキュリティ環境では、組織は多くの場合：

- ユーザーBYOKを完全に無効化します。
- データの居住地と監査ログを実行するゲートウェイ経由ですべてのトラフィックをルーティングします。

## MCPサーバー

**Model Context Protocol（MCP）**により、Droidは外部システム—チケットキュー、ドキュメント、データベースなど—に明確に定義されたツールを通してアクセスできます。

MCPサーバーは非常に強力であり、内部システムからの読み取りや副作用のあるアクションを実行する可能性があります。Factoryではそれらを制御するためのいくつかの手段を提供します：

- **組織許可リスト/ブロックリスト**
  - 組織管理者は、どのMCPサーバーがまったく許可されるかを定義します。
  - 許可リストにないサーバーは、プロジェクトが設定しようとしても無視されます。

- **プロジェクトレベル設定**
  - プロジェクトは許可されたサーバーのサブセットを有効化し、環境変数と接続の詳細を設定できます。

- **ユーザーレベルのオプトイン**
  - ユーザーは自分のセッションで、許可されたセットからMCPサーバーを有効または無効にできます。

これらの制御により、各サーバーがセキュリティとコンプライアンスのレビューを通過していることを確保しながら、内部システムをDroidに安全に公開できます。

## ドロイド、コマンド、ツール

Factoryの**ドロイド、コマンド、フック**のエコシステムも階層的に管理されます。

### 組織レベルのドロイドとコマンド

- 組織管理者は、**承認済みドロイド**と**共有コマンド**を組織レベルの`.factory`バンドルに公開できます。
- これらは多くの場合、以下のようなセキュリティレビュー済みワークフローをエンコードします：
  - `/security-review`
  - `/migrate-service`
  - `/refactor-module`
- プロジェクトとユーザーはこれらのリソースを使用できますが、変更はできません。

### プロジェクトレベルの拡張

- プロジェクトは、独自の`.factory/droids/`と`.factory/commands/`ディレクトリに**専用のドロイドとコマンド**を追加します。
- これらは、特定のサービス、スキーマ、ランブックの知識など、プロジェクト固有のロジックで組織リソースを拡張します。

### ユーザーレベルのカスタマイズ

- ユーザーは`~/.factory/droids/`と`~/.factory/commands/`に個人的なドロイドとコマンドを追加できます。
- これらは個人のワークフローに有用ですが、組織とプロジェクトのポリシーを依然として遵守する必要があります（例：不許可のツールやモデルを呼び出すことはできません）。

フックは、これらのツールの境界でエンフォースメントとロギングを提供することで、すべてを結びつけます。詳細は[LLMセーフティとエージェント制御](/jp/enterprise/llm-safety-and-agent-controls)を参照してください。

## 統合環境

DroidはCLIファーストであるため、開発者を特定のIDEに縛ることなく、多くの環境と統合します。

一般的なパターンには以下があります：

- **ターミナルとシェル**
  - `bash`、`zsh`、`fish`、またはWindowsシェルでのDroidの直接使用。
  - チーム間でワークフローを標準化するためのシェルエイリアスとスクリプト。

- **IDEとエディター統合**
  - VS Code、JetBrains ツールなど、Droidをバックエンドエージェントとして扱うIDEとの統合。
  - ポリシーとテレメトリは依然として同じ階層設定とOTELパイプラインを通して流れます。

- **CI/CDパイプライン**
  - GitHub Actions、GitLab CI、または内部パイプラインでコードレビュー、リファクタリング、移行などのタスクでDroidを実行。
  - 開発者とは別のIDと認証情報をCIで使用。

- **リモートと制限された環境**
  - エアギャップまたは制限された環境でのDroid実行。
  - セキュアな企業デプロイメントのためのデスクトップとブラウザベースインターフェースのサポート。

これらすべてにおいて、同じ企業制御が適用されます：**モデル、ツール、MCPサーバー、テレメトリは、IDEや環境ではなく、組織とプロジェクトのポリシーによって制約されます**。
