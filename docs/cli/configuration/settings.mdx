Configure Droid with `/settings` for interactive changes or `.factory` JSON files for repeatable team configuration. The current settings surface is schema-backed JSON under `.factory`.

## Open settings in Droid

1. Run `droid` in a terminal.
2. Enter `/settings`.
3. Change the setting you want.

Droid saves interactive changes to your user settings file and applies them to new sessions. Some changes, such as model selection, can also apply to the active session immediately.

## Settings file locations

| Scope | Path | Use |
| :---- | :--- | :-- |
| User | `~/.factory/settings.json` | Personal defaults across projects. |
| User local override | `~/.factory/settings.local.json` | Machine-specific personal overrides. |
| Project | `<project>/.factory/settings.json` | Shared project defaults committed with the repository. |
| Project local override | `<project>/.factory/settings.local.json` | Machine-specific project overrides. |
| Nested folder | `<folder>/.factory/settings.json` | Defaults for a subtree inside a larger repository. |

Settings files accept JSON with comments. Add `settings.local.json` to `.gitignore` if it contains machine-specific values.

## How settings are resolved

Droid resolves settings from organization controls, runtime/session options, folder and project `.factory` files, user settings, dynamic defaults, and built-in defaults. More specific local session defaults can override broader defaults, while organization controls can still restrict or cap choices such as model access, MCP tools, command policy, and maximum Autonomy Level.

Use `/settings-debug` when you need to see which source supplied a resolved setting.

## Configure session defaults

Use `sessionDefaultSettings` to set the model, reasoning effort, interaction mode, Autonomy Level, and Spec Mode defaults for new sessions.

```json
{
  "sessionDefaultSettings": {
    "model": "MODEL_ID",
    "reasoningEffort": "medium",
    "interactionMode": "auto",
    "autonomyLevel": "low"
  }
}
```

| Key | Values | Description |
| :-- | :----- | :---------- |
| `model` | Any available model ID | Default model for new sessions. Use `/model` or [Available Models](/models) to choose a current ID. |
| `reasoningEffort` | `off`, `none`, `low`, `medium`, `high` | Default reasoning effort. Droid clamps or defaults this value when the selected model does not support it. |
| `interactionMode` | `auto`, `spec` | Start in Auto or Spec Mode. |
| `autonomyLevel` | `off`, `low`, `medium`, `high` | Controls [what Droid can do without asking first](/cli/user-guides/auto-run). |
| `specModeModel` | Any available model ID | Optional model for Spec Mode planning. Falls back to `model` when unset. |
| `specModeReasoningEffort` | `off`, `none`, `low`, `medium`, `high` | Optional reasoning effort for the Spec Mode model. |

<Note>
  `sessionDefaultSettings.autonomyMode` is deprecated and kept only for older configurations. Use `interactionMode` plus `autonomyLevel` for new settings.
</Note>

## Configure Spec Mode defaults

Set a dedicated Spec Mode model when you want planning and implementation to use different model settings.

```json
{
  "sessionDefaultSettings": {
    "model": "MODEL_ID",
    "reasoningEffort": "medium",
    "specModeModel": "SPEC_MODEL_ID",
    "specModeReasoningEffort": "high"
  }
}
```

If `specModeModel` is missing, Spec Mode uses the main session model. If `specModeReasoningEffort` is missing, Droid uses the reasoning default for the effective Spec Mode model.

## Configure command policy

Use command policy to define commands that should run without extra confirmation and commands that should always require confirmation or be blocked by policy.

```json
{
  "commandAllowlist": ["ls", "pwd", "git status"],
  "commandDenylist": ["rm -rf /", "mkfs", "shutdown"]
}
```

| Key | Description |
| :-- | :---------- |
| `commandAllowlist` | Commands treated as low-risk for your workflow. |
| `commandDenylist` | Commands that should not run automatically. Denylist entries take precedence over allowlist entries. |

Any command not covered by these lists falls back to the active Autonomy Level and command restrictions.

## Configure custom models and BYOK

Add custom models with `customModels`. Each entry describes the model ID, provider, endpoint, API key, display name, limits, reasoning/thinking settings, optional headers, optional provider arguments, and image-support behavior.

```json
{
  "customModels": [
    {
      "model": "provider/model-name",
      "baseUrl": "https://api.example.com/v1",
      "apiKey": "YOUR_API_KEY",
      "provider": "generic-chat-completion-api",
      "displayName": "Example Custom Model",
      "maxContextLimit": 200000,
      "maxOutputTokens": 16000,
      "extraHeaders": {
        "X-Example-Header": "example-value"
      }
    }
  ]
}
```

See [Bring Your Own Key](/cli/byok/overview) for provider setup details.

## Configure hooks, MCP, commands, skills, and droids

`.factory` can also hold reusable automation and extension files:

| Surface | File or directory | Configure |
| :------ | :---------------- | :-------- |
| Hooks | `hooks/hooks.json` or `settings.json` `hooks` | Hook events, commands, `hooksDisabled`, and `showHookOutput`. See [Hooks](/cli/configuration/hooks-guide). |
| MCP servers | `mcp.json` | MCP server definitions, disabled state, and disabled tools. See [MCP](/cli/configuration/mcp). |
| Custom slash commands | `commands/` | Project or user slash commands. |
| Skills | `skills/` | Reusable workflows and context. |
| Custom droids | `droids/` | Subagent definitions. |
| Plugins | plugin registry files under `.factory` | Installed plugin bundles and marketplaces. |

## Configure display, audio, timeouts, and compaction

Common personal preferences live at the root of `settings.json`.

| Key | Description |
| :-- | :---------- |
| `diffMode` | Diff rendering mode, such as `github` or `unified`. |
| `theme`, `logoAnimation`, `overrideTerminalColors` | Terminal display preferences. |
| `completionSound`, `awaitingInputSound`, `soundFocusMode` | Audio cues and when they play. |
| `showThinkingInMainView` | Whether reasoning blocks appear in the main chat view. |
| `statusLine` | Custom status line command. |
| `compactionTokenLimit` | Context compaction controls. |
| `cloudSessionSync`, `includeCoAuthoredByDroid`, `enableDroidShield`, `ideAutoConnect` | Session sync, commit attribution, Droid Shield, and IDE connection defaults. |
