Use this quickstart to install the Droid CLI, inspect a project, choose the right planning and autonomy controls, and make your first change.

## Before you begin

Make sure you have:

- A terminal open in a code project
- Git repository (recommended for full workflow demonstration)

## Step 1: Install and start the Droid CLI

<CodeGroup>

```bash macOS/Linux
curl -fsSL https://app.factory.ai/cli | sh
```

```bash Homebrew
brew install --cask droid
```

```powershell Windows
irm https://app.factory.ai/cli/windows | iex
```

```bash npm
npm install -g droid
```

</CodeGroup>

<Note>
  **Linux users:** Ensure `xdg-utils` is installed for proper functionality. Install with: `sudo apt-get install xdg-utils`
</Note>

Then navigate to your project and start the Droid CLI.

<CodeGroup>
```bash
# Navigate to your project
cd /path/to/your/project

# Start interactive session
droid
```

</CodeGroup>

You'll see Droid's welcome screen in a full-screen terminal interface. If prompted, sign in via your browser to connect your Factory account.

## Step 2: Explore your codebase

Start by having Droid understand your project. Try one of these questions:

```

> analyze this codebase and explain the overall architecture

```

```

> what technologies and frameworks does this project use?

```

```

> where are the main entry points and how is testing set up?

```

Droid reads your files contextually and uses available organization context to explain project structure, conventions, and relevant files.

## Step 3: Choose Spec Mode and an Autonomy Level

Before you ask Droid to change code, choose how much planning and automation you want:

- **Spec Mode** plans before implementation. Use it for multi-file features, risky changes, migrations, and tasks where you want to approve the approach first.
- **Autonomy Level Off** keeps manual approval for file edits and commands. This is the safest first-run setting.
- **Autonomy Level options Low, Medium, and High** let Droid proceed without repeated confirmations for work at or below the selected risk level.

Press **Shift+Tab** to switch between Auto and Spec Mode. Press **Ctrl+L** to change Autonomy Level, or keep it Off for manual approvals. See [Spec Mode](/cli/user-guides/specification-mode) and [Autonomy Level](/cli/user-guides/auto-run) for the full controls.

## Step 4: Make your first code change

Now try a simple modification:

```

> add comprehensive logging to the main application startup

```

Droid will:

1. Analyze your current logging setup
2. Propose a concrete approach
3. Show you what will be modified
4. Apply changes according to your selected Autonomy Level and approval prompts

Keep Autonomy Level Off for your first run if you want Droid to ask before each edit and command.

## Step 5: Experience enterprise workflows

Try a more complex task that showcases how Droid uses integrated context:

```

> audit this codebase for security vulnerabilities and create a remediation plan

```

Or provide context from your team's tools by pasting a link:

```

> implement the feature described in this Jira ticket: https://company.atlassian.net/browse/PROJ-123

```

If you've integrated these platforms through Factory's dashboard, Droid can read context from Jira, Notion, Slack, and other sources. Droid can also connect to additional tools through MCP integrations.

## Step 6: Handle version control

Droid makes Git operations conversational and intelligent:

```

> review my uncommitted changes and suggest improvements before I commit

```

```

> create a well-structured commit with a descriptive message following our team conventions

```

```

> analyze the last few commits and identify any potential issues or patterns

```

## Essential controls

Here are the key interactions you'll use daily:

| Action             | What it does                                 | How to use                               |
| ------------------ | -------------------------------------------- | ---------------------------------------- |
| Send message       | Submit a task or question                    | Type and press **Enter**                 |
| Multi-line input   | Write longer prompts                         | **Shift\+Enter** for new lines           |
| Enter Spec Mode    | Plan and approve before implementation       | Press **Shift\+Tab** until Spec Mode     |
| Set Autonomy Level | Choose Off, Low, Medium, or High permissions | Press **Ctrl\+L**                       |
| Approve changes    | Accept proposed modifications                | Accept change in the TUI                 |
| Reject changes     | Decline proposed modifications               | Reject change in the TUI                 |
| View shortcuts     | See all available commands                   | Press **?**                              |
| Exit session       | Leave Droid                                  | **Ctrl\+C** or type `exit`               |

### Useful slash commands

Quick shortcuts to common actions:

- `/review` - Start AI-powered code review workflow ([learn more](/cli/features/code-review))
- `/settings` - Configure Droid behavior, models, and preferences
- `/model` - Switch between AI models mid-session
- `/mcp` - Manage Model Context Protocol servers
- `/readiness-report` - Generate an Agent Readiness report for the current repository
- `/readiness-fix` - Fix issues identified by Agent Readiness checks
- `/account` - Open your Factory account settings in browser
- `/billing` - View and manage your billing settings
- `/help` - See all available commands

[Learn how to create custom slash commands →](/cli/configuration/custom-slash-commands)

## Collaboration best practices

**Be specific with context:**
Instead of: "fix the bug"
Try: "fix the authentication timeout issue where users get logged out after 5 minutes instead of the configured 30 minutes"

**Use Spec Mode for complex features:**
For larger features, switch to [Spec Mode](/cli/user-guides/specification-mode) before sending the request so Droid researches and proposes a plan before implementation.

**Pick the right Autonomy Level:**
Use Off or Low for unfamiliar work, Medium for routine local development, and High only when the environment is safe for broader automation.

**Leverage organizational knowledge:**

```

> following our team's coding standards, implement the user preferences feature described in ticket PROJ-123

```

**Use the review workflow:**
Review Droid's proposed changes before approval. The transparent diff view helps you understand exactly what will be modified.

## Pro tips for enterprise teams

**Security-first approach:**
Droid considers security implications and can flag potential vulnerabilities during code generation.

**Compliance integration:**
Connect your compliance tools through MCP so Droid can use them during implementation and verification.

**Team knowledge sharing:**
Droid uses your organization's patterns to help maintain consistency across team members and projects.
