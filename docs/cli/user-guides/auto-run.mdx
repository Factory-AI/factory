Autonomy Level sets the highest-risk work Droid can run without pausing for approval. It is separate from interaction mode: Auto executes work, while Spec Mode plans before implementation.

## Choose a level

Execute commands and MCP tools have a risk level (`low`, `medium`, or `high`). Droid runs them automatically when the risk is at or below your Autonomy Level, unless a denylist or sandbox check requires approval.

| Autonomy Level | What can run without approval                           | Examples                                                               |
| -------------- | ------------------------------------------------------- | ---------------------------------------------------------------------- |
| **Off**        | Built-in read tools and allowlisted commands only       | `Read`, `LS`, `ls`, `pwd`, `git status`                                |
| **Low**        | File edits plus low-risk commands and MCP tools         | `Edit`, `Create`, `rg`, showing logs                                   |
| **Medium**     | Everything from Low plus reversible workspace changes   | `npm install`, `pip install`, `git commit`, `mv`, `cp`, build tooling  |
| **High**       | High-risk actions unless safety checks require approval | `docker compose up`, `git push` if allowed, migrations, custom scripts |

Droid still streams output and highlights file changes at every level.

## Tool availability vs. approval

Autonomy Level controls automatic approval. It is not a complete list of every tool Droid can request.

- **Interactive CLI** – Tools can still be requested when they exceed the current Autonomy Level; Droid pauses for approval before using them.
- **Droid Exec** – Without `--auto`, Droid Exec exposes read-only tools plus `Execute` for low-risk commands. Any `--auto` level enables mutation tools, while `Task` subagent orchestration requires `--auto high` or `--skip-permissions-unsafe` at the top level.
- **Tool policy** – Model support, custom droid tool lists, `--enabled-tools`, `--disabled-tools`, MCP server configuration, and organization controls can further restrict which tools are available.

## How approvals work

- **Auto vs. Spec Mode** – Autonomy Level applies while Droid is in Auto. Spec Mode remains read-only except for saved spec files.
- **File changes** – Low or higher lets Droid create, edit, and patch files without asking first.
- **Commands and MCP tools** – Droid compares the tool risk level to your Autonomy Level. If the risk is higher, it asks before continuing.
- **Allowlisted commands** – Commands in the allowlist can run without approval unless they also match the denylist.
- **Spec approval** – When approving a Spec Mode plan, choose **Proceed with implementation** to keep manual controls, or choose an available Low, Medium, or High option for the implementation phase. Organization Maximum Autonomy Level can hide higher options.
- **Safety checks** – Denylisted dangerous commands still ask at High, including dangerous commands nested inside `$(...)` or backticks. Sandbox read, write, and network checks can also prompt separately.
- **Allow always** – Choosing an “always allow” option raises the current Autonomy Level to the level required by that prompt. Sandbox “allow always” options instead persist the allowed path or domain.

## Command allowlists and denylists

Use `commandAllowlist` and `commandDenylist` in [Settings](/cli/configuration/settings) to encode command policy for your user profile, a project, a local project override, or a nested folder.

- Allowlist entries are treated as low-risk for the matching scope.
- Denylist entries always take precedence over allowlist entries.
- Commands not covered by either list fall back to the active Autonomy Level and command restrictions.
- Organization-managed settings have the highest priority. Local and project settings can add defaults for a repo or machine, but they cannot weaken organization command policy or raise autonomy above the organization maximum. See [Hierarchical Settings & Org Control](/enterprise/hierarchical-settings-and-org-control).

The default denylist includes these command patterns:

```json
[
  "rm -rf /",
  "rm -rf /*",
  "rm -rf .",
  "rm -rf ~",
  "rm -rf ~/*",
  "rm -rf $HOME",
  "rm -r /",
  "rm -r /*",
  "rm -r ~",
  "rm -r ~/*",
  "mkfs",
  "mkfs.ext4",
  "mkfs.ext3",
  "mkfs.vfat",
  "mkfs.ntfs",
  "dd if=/dev/zero of=/dev",
  "dd of=/dev",
  "shutdown",
  "reboot",
  "halt",
  "poweroff",
  "init 0",
  "init 6",
  ":(){ :|: & };:",
  ":() { :|:& };:",
  "chmod -R 777 /",
  "chmod -R 000 /",
  "chown -R",
  "Format-Volume",
  "format.com",
  "powershell Remove-Item -Recurse -Force"
]
```

## Change the level

- Press `Ctrl+L` to cycle `Off → Low → Medium → High → Off`. Organization policy can cap the highest available level.
- Press `Shift+Tab` to switch between Auto and Spec Mode. On Windows or WSL, use `Ctrl+T` if `Shift+Tab` is unavailable.
- Set a default in the Factory CLI settings menu for future sessions.
- Change Autonomy Level before implementation, from the [Spec Mode](/cli/user-guides/specification-mode) approval dialog, or any time after leaving Spec Mode.

## Where Autonomy Level applies

- **Interactive CLI** – `droid` uses the session's current Autonomy Level. `droid "<prompt>"` starts the same interactive CLI with an initial prompt, so the first task uses your configured default. See the [CLI reference](/reference/cli-reference).
- **Droid Exec** – `droid exec` is read-only by default. Use `--auto low`, `--auto medium`, or `--auto high` for non-interactive runs that need edits, local development commands, or broader automation. See [Droid Exec](/cli/droid-exec/overview).
- **Custom Droids (Subagents)** – subagents run as child Droid Exec sessions. In Auto, Task-launched subagents request `--auto high` regardless of the parent session's current Autonomy Level; the organization Maximum Autonomy Level can still cap that, and the droid tool policy can limit available tools. In Spec Mode, subagents are restricted to read-only work. See [Custom Droids](/cli/configuration/custom-droids).
- **Missions** – Mission orchestration uses high autonomy for the orchestrator and worker sessions. `droid exec --mission` requires `--auto high` or `--skip-permissions-unsafe`, and admins can restrict who can start Missions. See [Missions](/cli/features/missions).

## Enterprise controls

Enterprise admins can set organization-wide autonomy boundaries with organization-managed settings. See [Hierarchical Settings & Org Control](/enterprise/hierarchical-settings-and-org-control).

- **Default Autonomy Level** sets the starting level for new sessions.
- **Maximum Autonomy Level** caps how high members can raise autonomy. If the maximum is Medium, High is unavailable in the CLI.

These controls layer with command allowlists, command denylists, MCP restrictions, sandbox settings, and Mission access controls.

## Use it safely

- Start new or high-stakes work with Off or Low until you trust the plan.
- Match the minimum level to the work: use Low for file edits and generated reports, Medium when the run must install dependencies, build, test, or make local commits, and High for pushes, deployments, Task-launched subagents, Missions, or other orchestration.
- Use Medium for day-to-day feature development and refactors that rely on package managers or build steps.
- Reserve High for well-understood pipelines in safe environments.
- Add defense in depth for automation: use [blocking hooks](/cli/configuration/hooks-guide) for repository-specific command checks, keep command denylists and MCP restrictions in organization-managed settings, and harden the host with least-privilege credentials, isolated runners, and network egress controls.
- For action-based code review and other CI workflows, choose the lowest `droid exec --auto` level that allows the workflow to complete. Review-only or report-writing workflows often need Low; workflows that run installs, tests, commits, pushes, or subagents may need Medium or High. See [Automated Code Review](/guides/droid-exec/code-review) and [GitHub Actions examples](/guides/droid-exec/github-actions).
- If you spot a suspect command, interrupt, provide guidance, and resume at the Autonomy Level that fits the remaining risk.
