Spec Mode turns a high-level request into a reviewed specification and implementation plan before Droid edits files or runs modifying commands. Use it for multi-file changes, migrations, high-risk work, or any task where you want to shape the approach before implementation starts.

## How it works

<Steps>
  <Step title="Describe your feature">
    Provide a simple description in 4-6 sentences. Focus on the outcome,
    constraints, and success criteria.
  </Step>
  <Step title="Droid creates the spec">
    Droid analyzes your request and generates a complete specification with
    acceptance criteria, implementation plan, and technical details. If the
    request is ambiguous, Droid can ask clarifying questions before finalizing
    the plan.
  </Step>
  <Step title="Review and approve">
    You review the generated specification and implementation plan. Request
    changes or approve as-is.
  </Step>
  <Step title="Implementation">
    After approval, Droid exits Spec Mode and begins making code changes
    according to the plan. The current or selected Autonomy Level controls
    implementation approvals.
  </Step>
</Steps>

## Example workflow

**Your input:**

```
Add a feature for users to export their personal data.
It should create a ZIP file with their profile, posts, and uploaded files.
Send them an email when it's ready. Make sure it follows GDPR requirements.
The export should work for accounts up to 2GB of data.
```

**Droid generates:**

- Complete specification with detailed acceptance criteria
- Technical implementation plan covering backend, frontend, and email
- File-by-file breakdown of changes needed
- Testing strategy and verification steps
- Security and compliance considerations

**You approve, then Droid implements** the complete solution while surfacing changes according to the current or selected Autonomy Level.

<Note>
  Spec Mode must be activated before the request you want planned. It does not
  automatically activate for every complex prompt.
</Note>

## How to activate Spec Mode

In the interactive Factory CLI, press **Shift+Tab** until the status banner shows Spec Mode. Send your request after Spec Mode is active.

Use **Ctrl+L** separately to change [Autonomy Level](/cli/user-guides/auto-run). Spec Mode and Autonomy Level are independent controls.

In the Desktop App, select **Spec Mode** from the mode or model controls before sending your request.

In Droid Exec, pass `--use-spec`:

```bash
droid exec --use-spec "add user profiles"
```

## What happens during planning

**Analysis phase (read-only):**

- Examines your existing codebase and patterns
- Reviews related files and dependencies
- Studies your AGENTS.md conventions
- Gathers context from external sources you provide

**Planning phase:**

- Develops a comprehensive implementation strategy
- Identifies files that need changes
- Plans the sequence of modifications
- Defines testing and verification steps

**Safety guarantees:**

- Cannot edit files during analysis
- Cannot run commands that modify anything
- Cannot create, delete, or move files
- Exploration stays read-only until you approve

## When to use Spec Mode

Use Spec Mode when the change is larger than a quick edit or when you want Droid to reason through the implementation before touching code.

- Plan new features or large refactors.
- Turn a high-level request into a detailed specification.
- Add safety and correctness checks for complex changes.
- Get a clear implementation roadmap before any files are modified.

For small, well-understood edits, Auto can proceed directly while still using your current Autonomy Level and approval prompts.

## Writing effective requests

**Focus on outcomes:** Describe what the software should accomplish, not how to build it.

```
Users need to be able to reset their passwords using email verification.
The reset link should expire after 24 hours for security.
Include rate limiting to prevent abuse.
```

**Include important constraints:**

```
Add user data export functionality that works for accounts up to 5GB.
Must comply with GDPR and include audit logging.
Should complete within 10 minutes and not impact application performance.
```

**Reference existing patterns:**

```
Add a notification system similar to how we handle email confirmations.
Use the same background job pattern as our existing report generation.
Follow the authentication patterns we use for other sensitive operations.
```

**Be specific about verification:** Tell Droid how to confirm the implementation works correctly.

**Consider the full user journey:** Describe the complete experience, not just technical requirements.

**Include error scenarios:** Specify how failures should be handled and communicated to users.

**Think about scale:** Mention performance requirements and expected usage patterns.

## Enterprise integration

Reference external requirements by pasting links:

```
Implement the user management features described in this Jira ticket:
https://company.atlassian.net/browse/PROJ-123

Follow our security standards and include comprehensive error handling.
```

If you've integrated platforms through Factory's dashboard, Droid can read context from tickets, documents, and specs during analysis.

## AGENTS.md integration

Document your project conventions to improve Spec Mode planning. See [AGENTS.md](/cli/configuration/agents-md) for more information.

Spec Mode incorporates these conventions so plans align with your team's standards.

## Using mixed models

Open `/model` and configure a dedicated Spec Mode model when planning should use a different model than implementation. A stronger model can produce higher-quality specs for complex work, while a faster or lower-cost default model can handle implementation once the plan is approved.

If no Spec Mode model is configured, Spec Mode uses the main session model. See [Mixed Models](/cli/configuration/mixed-models) for setup details.

## Breaking down large features

For complex features spanning multiple components, break them into focused phases:

**Phase 1:**

```
Implement user data export backend API and job processing.
Focus only on the server-side functionality, not the UI yet.
```

**Phase 2:**

```
Add the frontend UI for data export using the API from Phase 1.
Include progress indicators and download management.
```

This approach lets you validate each phase before proceeding to the next.

## Approval options

After Droid presents the plan, choose how to continue:

1. **Proceed with implementation** – Approve the plan, exit Spec Mode, and keep manual execution controls.
2. **Proceed, and allow file edits and read-only commands (Low)** – Approve the plan and use Autonomy Level Low for implementation.
3. **Proceed, and allow reversible commands (Medium)** – Approve the plan and use Autonomy Level Medium for implementation.
4. **Proceed, and allow all commands (High)** – Approve the plan and use Autonomy Level High for implementation.
5. **No, keep iterating on spec** – Stay in Spec Mode to refine the plan before implementation.

Organization Maximum Autonomy Level can hide higher approval options. Choosing an Autonomy Level option sets that level for implementation; choosing to keep iterating leaves Spec Mode active.

## Saving your specifications as Markdown

Spec Mode can automatically write the approved plan to disk. Open the Factory CLI settings and enable **Save spec as Markdown** to turn this on.

- By default, plans are saved to `.factory/docs` inside the nearest project-level `.factory` directory. If none exists, the Factory CLI falls back to `~/.factory/docs` in your home directory.
- Use the **Spec save directory** setting to pick between the project directory, your home directory, or a custom path. Custom values support absolute paths, `~` expansion, `.factory/...` shortcuts, and relative paths from the current workspace.
- The Factory CLI creates the target directory if it does not exist and writes the Markdown exactly as shown in the approval dialog.
- Files are named `YYYY-MM-DD-slug.md`, where the slug comes from the spec title or first heading, and a counter is appended if a file with the same name already exists.

## What happens after approval

Once you approve a plan, Droid systematically implements the changes according to the current or selected Autonomy Level. You can keep manual approval on, raise autonomy for safe automation, or interrupt if the work needs a course correction.

For simpler changes that don't need comprehensive planning, Droid can proceed directly while still showing changes according to your selected controls.
