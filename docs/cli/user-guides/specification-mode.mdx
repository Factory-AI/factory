Spec Mode turns a high-level request into a reviewed specification and implementation plan before Droid edits files or runs modifying commands. Use it for multi-file changes, migrations, high-risk work, or any task where you want to shape the approach before implementation starts.

## How it works

<Steps>
  <Step title="Describe your feature">
    Provide a simple description in 4-6 sentences. Focus on the outcome,
    constraints, and success criteria.
  </Step>
  <Step title="Droid creates the spec">
    Droid analyzes your request and generates a complete specification with
    acceptance criteria, implementation plan, and technical details. If the
    request is ambiguous, Droid can ask clarifying questions before finalizing
    the plan.
  </Step>
  <Step title="Review and approve">
    You review the generated specification and implementation plan. Request
    changes or approve as-is.
  </Step>
  <Step title="Implementation">
    After approval, Droid exits Spec Mode and begins making code changes
    according to the plan. The current or selected Autonomy Level controls
    implementation approvals.
  </Step>
</Steps>

<Note>
  Spec Mode must be activated before the request you want planned. It does not
  automatically activate for every complex prompt.
</Note>

## How to activate Spec Mode

In the CLI, press **Shift+Tab** until the status banner shows Spec Mode. Send your request after Spec Mode is active.

Use **Ctrl+L** separately to change [Autonomy Level](/cli/user-guides/auto-run). Spec Mode and Autonomy Level are independent controls.

In the Desktop App, select **Spec Mode** from the mode or model controls before sending your request.

In Droid Exec, pass `--use-spec`:

```bash
droid exec --use-spec "add user profiles"
```

## What happens during planning

**Analysis phase (read-only):**

- Examines your existing codebase and patterns
- Reviews related files and dependencies
- Studies your AGENTS.md conventions
- Gathers context from external sources you provide

**Planning phase:**

- Develops a comprehensive implementation strategy
- Identifies files that need changes
- Plans the sequence of modifications
- Defines testing and verification steps

**Safety guarantees:**

- Cannot edit files during analysis
- Cannot run commands that modify anything
- Cannot create, delete, or move files
- Exploration stays read-only until you approve

## When to use Spec Mode

Use Spec Mode when the change is larger than a quick edit or when you want Droid to reason through the implementation before touching code.

- Plan new features or large refactors.
- Turn a high-level request into a detailed specification.
- Add safety and correctness checks for complex changes.
- Get a clear implementation roadmap before any files are modified.

For small, well-understood edits, Auto can proceed directly while still using your current Autonomy Level and approval prompts.

## Writing effective requests

- Focus on outcomes. Describe what the software should accomplish, not how to build it.
- Include important constraints
- Reference existing patterns.
- Be specific about verification. Tell Droid how to confirm the implementation works correctly.
- Consider the full user journey. Describe the complete experience, not just technical requirements.
- Include error scenarios. Specify how failures should be handled and communicated to users.
- Think about scale. Mention performance requirements and expected usage patterns.

## Using mixed models

Open `/model` and configure a dedicated Spec Mode model when planning should use a different model than implementation. A stronger model can produce higher-quality specs for complex work, while a faster or lower-cost default model can handle implementation once the plan is approved.

If no Spec Mode model is configured, Spec Mode uses the main session model. See [Mixed Models](/cli/configuration/mixed-models) for setup details.

## Saving your specifications as Markdown

Spec Mode can automatically write the approved plan to disk. Open Settings and enable **Save spec as Markdown** to turn this on.

- By default, plans are saved to `.factory/docs` inside the nearest project-level `.factory` directory. If none exists, the Droid falls back to `~/.factory/docs` in your home directory.
- Use the **Spec save directory** setting to pick between the project directory, your home directory, or a custom path. Custom values support absolute paths, `~` expansion, `.factory/...` shortcuts, and relative paths from the current workspace.
- Droid creates the target directory if it does not exist and writes the Markdown exactly as shown in the approval dialog.
- Files are named `YYYY-MM-DD-slug.md`, where the slug comes from the spec title or first heading, and a counter is appended if a file with the same name already exists.
