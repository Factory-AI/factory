Factory operates a dedicated EU deployment for organizations with European data-residency requirements. EU-tagged organizations have their session content stored in the EU, their LLM inference dispatched to EU-region model endpoints, and their traffic served by a separate EU backend that is fully isolated from the Global (US) deployment.

The same Factory CLI, desktop app, and web app are used regardless of region; clients detect the organization's region after login and automatically route to the correct deployment.

---

## Infrastructure regions

The EU deployment runs on dedicated infrastructure pinned to two European regions:

- **EU backend (API and web app):** Hosted on Vercel in the Frankfurt region (`fra1`). All `app.eu.factory.ai` and `api.eu.factory.ai` traffic terminates here, and no requests for EU organizations are served by the Global (US) backend.
- **EU database:** A managed database instance in Belgium (`europe-west1`), holding all session content for EU organizations. The database is provisioned separately from the Global database, with its own credentials, backups, and access controls.

Both regions are within the European Economic Area, and session traffic stays within the EU.

---

## Data residency

**Stored in the EU (`europe-west1`, Belgium):**

- Session content: prompts, assistant messages, tool calls and results, droid messages, git AI notes, and git AI checkpoints. Stored in a dedicated EU managed database with point-in-time recovery, 7-day backups, and delete protection enabled.
- Raw request and response bodies are never written to logs or object storage on the EU deployment.

**Stored in the Global deployment (US):**

- Organization records (a small pointer document containing the org's region tag), user profiles, and billing data. These contain no user-generated content and are required for cross-region authentication and account management.

## LLM inference

All LLM requests from EU organizations are dispatched to EU-region endpoints only. Request payloads (including prompts and conversation history) do not transit US infrastructure during inference.

## Access and onboarding

**Entry points:**

- Web app: `https://app.eu.factory.ai`
- API: `https://api.eu.factory.ai`

**Onboarding:** EU organizations are provisioned by Factory. Customers requiring an EU org should contact their Factory account team. Self-serve organization creation is not available on the EU deployment.

**Identity and SSO:** Factory uses WorkOS-based SSO. Identity is global, but each organization is pinned to a region at creation time. After authentication, only that org's region-resident data is loaded.

**Network allowlists:** Customers should allowlist `*.factory.ai` hostnames. Outbound LLM traffic from the EU backend targets AWS `eu-central-1`, GCP `europe-west1`, and OpenAI's EU-residency endpoints.

## Region enforcement and client routing

Region matching is enforced at multiple layers so that an organization's data and inference traffic can never be served by the wrong backend, even if a client mis-targets.

### Server-side enforcement

Both session data access and LLM inference are gated by region-aware checks on the backend. Requests for an organization whose region does not match the deployment fail closed with an error, and requests for a model or provider that is not eligible for the deployment region are rejected before any traffic is dispatched. EU traffic cannot reach US-only data or providers, and Global traffic cannot reach EU-only routes.

### Client-side routing

The same Factory CLI, desktop app, and web app are used by EU and Global organizations. After login, each client reads the organization's region and routes all subsequent API requests to the matching backend:

- **Web app:** Each canonical hostname (`app.factory.ai`, `app.eu.factory.ai`) is statically backed by exactly one regional backend. Users who land on the wrong region are shown a redirect page with a link to the correct origin; they are not signed out.
- **Desktop app:** Reads the org's region from the authenticated session and targets the matching API host on every request.
- **CLI:** The region is pinned to the user's local token cache on first login and used to resolve the API host for every request. Logout and re-login automatically re-derive the region.
