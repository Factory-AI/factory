# Docs style

Style rules here exist to prevent common reader mistakes, not to enforce taste.

## Rules with rationale

- Lead with the task or product mechanism.
  Readers scan for the action they need, not a tour of the page.
- Use exact shipped names from nearby public docs.
  Generic labels blur product surfaces and create support ambiguity.
- Say what a control changes.
  `Manage policy` is vague; `restrict commands` tells the reader the effect.
- Cross-link only when it changes the next step.
  Repeated links add noise and make important paths harder to see.
- Avoid placeholder links.
  Public docs should not publish dead affordances or fake navigation.
- Describe screenshots by meaning.
  Alt text should explain the UI or diagram, not the filename.
- Keep changelog entries factual.
  Release notes are scanned for shipped behavior and regressions.
